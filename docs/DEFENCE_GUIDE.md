# Гайд по защите проекта SupervisorMatch

Этот документ предназначен для команды. Здесь кратко и по делу описана архитектура всего нашего проекта, как всё работает под капотом, и что нужно отвечать на защите, если будут вопросы по технической части.

## 1. Общая архитектура

Наш проект — это классическое клиент-серверное приложение (SPA — Single Page Application).
- **Frontend**: Написан на **React** (сборщик Vite). Отвечает только за отрисовку интерфейса и маршрутизацию (React Router).
- **Backend**: Написан на **NestJS** (TypeScript). Отвечает за всю бизнес-логику, безопасность, авторизацию и работу с данными.

Обе части общаются между собой по протоколу **HTTP/REST**. Формат обмена данными — **JSON**.

## 2. Аутентификация и безопасность (Как работает логин)

Если преподаватель спросит про безопасность, рассказываем про **JWT (JSON Web Token)**:
1. Пользователь вводит email и пароль.
2. Фронтенд отправляет их на `POST /auth/login`.
3. Бэкенд проверяет данные и, если всё верно, выдает 2 токена:
   - **Access Token** — короткоживущий токен. Бэкенд возвращает его в теле ответа (JSON). Фронтенд сохраняет его в `localStorage` и прикрепляет к каждому запросу в заголовке `Authorization: Bearer <token>`.
   - **Refresh Token** — долгоживущий токен. Бэкенд сам кладет его в защищенную куку (`httpOnly`), чтобы фронтенд-скрипты не имели к нему доступа. Это защищает нас от XSS-атак.
4. Когда Access Token протухает, фронтенд дергает `POST /auth/refresh`, и бэкенд (благодаря куке с Refresh токеном) выдает новый Access Token.

## 3. База данных и хранение

Мы используем полноценную реляционную базу данных **PostgreSQL**, которая хостится в облаке **Supabase**.
- **ORM (Object-Relational Mapping)**: Для работы с базой данных из кода мы используем **Prisma ORM**. Это современный и типобезопасный инструмент. Он позволяет нам не писать SQL-запросы вручную, а работать с базой через удобные методы (например, `prisma.user.findUnique()`), при этом автокомплит TypeScript гарантирует, что мы не ошибемся в названиях полей.
- **Масштабируемость**: Наш NestJS-сервер написан по паттерну "Repository". Логика базы данных изолирована от основной бизнес-логики. Мы успешно перенесли хранение данных с временных JSON-файлов (на этапе прототипа) на production-ready PostgreSQL, переписав только сервисы базы данных, не трогая контроллеры или фронтенд.

## 4. Как работает Фронтенд (Интеграция)

- **api.js**: В папке `src/lib/` лежит наш собственный клиент поверх `fetch`. Он автоматически перехватывает запросы, подставляет базовый URL (`http://localhost:4000`) и прикрепляет `Access Token`.
- **AppContext**: Сердце приложения. При загрузке страницы он берет токен из `localStorage` и отправляет запрос `GET /users/me`. Бэкенд узнает пользователя, и приложение "оживает" (восстанавливается сессия).
- В компонентах нет никаких "фейковых" данных. Каждый клик (добавить тему, отправить заявку, поменять статус) — это реальный асинхронный HTTP-запрос (`await api.post(...)`) на сервер.

## 5. Основные возможности (Что показывать на демо)

1. **Регистрация и Вход** (Студенты и Супервизоры).
2. **Просмотр профиля и редактирование** — запрос `PATCH /users/me` или `PATCH /supervisors/:id/profile`.
3. **Список супервизоров и тем** — Супервизор может добавлять/удалять темы (эндпоинты `POST` и `DELETE` по темам).
4. **Процесс заявки (Матчинг)**:
   - Студент видит список и свободные места (`capacity - currentStudents`).
   - Нажимает "Отправить запрос" -> летит `POST /requests`.
   - Супервизор видит запрос у себя в дашборде.
   - Супервизор нажимает "Принять" или "Отклонить" -> летит `PATCH /requests/:id/status`.
   - Количество свободных мест у супервизора пересчитывается на бэкенде.

## 6. Панель администратора (Admin Panel)

Панель администратора — это отдельная защищенная зона в приложении, предназначенная для управления системой (просмотр статистики, управление пользователями и заявками).

- **Как работает логин админа:** Учетная запись администратора **не хранится в базе данных**. Это сделано для повышения безопасности (system-level access). Логин и пароль задаются через переменные окружения (`ADMIN_LOGIN` и `ADMIN_PASSWORD` в файле `.env` бэкенда).
- **Безопасность админки:** При успешном входе выдается отдельный JWT-токен с ролью `super_admin`. Все запросы к маршрутам `/admin/*` на бэкенде защищены middleware `requireAdmin`, который проверяет этот токен.
- **Возможности:** Администратор может видеть всю статистику, список всех пользователей, а также напрямую подтверждать (Approve) или отклонять (Reject) заявки, минуя супервизоров. При этом профиль администратора редактируется (ФИО, email, аватар), но сохраняется только в памяти сервера (до перезагрузки), так как нет привязки к БД.

## 7. Запуск проекта (от А до Я)

Мы значительно упростили процесс локального запуска. Теперь и фронтенд, и бэкенд запускаются одной командой.

Перед запуском убедитесь, что в папке `backend` есть файл `.env` со следующими настройками:
```env
DATABASE_URL="postgres://..."
DIRECT_URL="postgres://..."
JWT_SECRET="super-secret-key"
JWT_ACCESS_SECRET="admin-secret-key"
ADMIN_LOGIN="admin"
ADMIN_PASSWORD="admin"
```

1. **Запуск базы данных и генерация клиента Prisma** (в папке `backend`):
   ```bash
   npx prisma db push
   npx prisma generate
   ```

2. **Запуск всего приложения** (в корневой папке проекта):
   ```bash
   npm install
   npm start
   ```
   Скрипт `npm start` использует пакет `concurrently` для одновременного запуска NestJS-бэкенда (на `http://localhost:4000`) и Vite-фронтенда (на `http://localhost:5173`) в одном окне терминала.

## 8. Деплой и Docker

Мы используем **Docker** для контейнеризации приложения. Это позволяет развернуть весь проект на любом сервере буквально одной командой.
В корне проекта есть `docker-compose.yml`, который описывает два сервиса:
- `backend` (Node.js сервер на порту 4000).
- `frontend` (React приложение, собранное через Vite и раздаваемое через веб-сервер на порту 80).

Чтобы запустить всё в докере на сервере, достаточно:
```bash
docker compose up -d --build
```
После этого фронтенд будет доступен по обычному IP-адресу или домену (порт 80), а бэкенд на порту 4000. Вся работа с базой данных (Supabase) происходит по сети, поэтому БД не требует контейнеризации.

## FAQ (Вопросы от комиссии)

- **Почему React?** Самая популярная библиотека, много готовых решений, компонентный подход ускоряет разработку интерфейсов.
- **Почему NestJS?** Строгая архитектура "из коробки" (модули, контроллеры, сервисы), полная поддержка TypeScript, легко масштабировать в будущем микросервисы.
- **Почему PostgreSQL и Prisma?** Реляционная модель отлично подходит для связей "Студент - Супервизор - Заявки". Prisma дает высокую скорость разработки, безопасность типов и удобные миграции. Облачная БД Supabase избавляет от необходимости администрировать свой сервер БД.
- **Почему учетка администратора не в базе данных?** Для обеспечения максимального уровня доступа (system-level) независимо от состояния базы данных или миграций. Это классический паттерн для master-аккаунтов.
- **А где стейт-менеджер (Redux, MobX)?** Для текущего масштаба приложения нам достаточно встроенного React Context API. Он отлично справляется с хранением сессии пользователя.
- **Как валидируете данные?** На бэкенде мы используем библиотеку Zod для валидации данных и типизации, а на фронте базовые проверки (длина пароля, пустые поля).
